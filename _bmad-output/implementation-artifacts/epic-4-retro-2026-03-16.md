## Ce qui a bien fonctionne

### Implementation en un bloc
L'Epic 4 a ete livree en un seul commit coherent. Infrastructure blog (Content Collections, BlogLayout, pages dynamiques) + 6 articles + navigation — tout d'un coup, sans iterations cassantes.

### Pattern anti-peurs efficace
Les 3 articles suivent le pattern defini : question SEO → reponse factuelle → Maison Ayaba → CTA. FAQ structurees pour Schema.org. Maillage interne vers les pages existantes.

### Fondations Epics 1-3 solides
BaseLayout, MetaTags, SchemaOrg, getAlternateUrl — tout reutilise sans friction pour le blog. L'investissement en architecture paie a chaque epic.

## Challenges identifies

### Contenu placeholder livre comme "fait"
L'Epic 4 a ete marquee "done" alors que les restaurants, commerces, POI etaient inventes. Le nom de l'hote etait "Onizuka" au lieu de "Guillaume". Les images blog etaient des placeholders. La validation du contenu reel a du etre faite manuellement par le projet lead.

### Sprint-status desynchronise
Le sprint-status montrait Epic 4 en backlog alors que le code etait deja committe. Manque de discipline sur la mise a jour du tracking.

### Header slice trop restrictif
La tech-spec prevoyait .slice(0, 5) pour inclure Blog, mais le code initial excluait encore Blog. Le fix a ete de retirer completement le slice.
